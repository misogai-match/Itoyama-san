1 今日起きたこと — 実例

3月30日 — 請求書未発行による入金遅延リスク

心ちゃん（小川）から、以下3件の請求書が未発行であることが報告された。

LINE報告内容

下記3つの請求書をちゃんと今日中に発行して、
明日入金してくれることも加味して、の残高になります…


DIA / 車両代金-島袋薫 ¥742,985 ※松尾対応
DIA / 車両代金-西澤博之 ¥786,057 ※小野寺対応
DIA / 車両代金-廣田久美 ¥952,140 ※村田対応

これが今日みたいに後出しで "ズレてしまいました" と言われたら、もっとマイナスになります

合計 ¥2,481,182 の入金が遅延するリスクがあった。
担当者が気づかず、心ちゃんが手動で発見して初めて対応が始まった。

2 構造的な問題 — なぜ繰り返されるか

① 担当者が気づかない

納車・陸送のスケジュール変更があっても、入金日への影響に気づかない。
日々の業務に追われ、入出金チェックシートの確認が後回しになる。

② 報告が漏れる

気づいたとしても、上への報告を忘れる・後回しにする。
「ズレてしまいました」の報告が事後になり、対応が後手に回る。

③ 情報がチャットに埋もれる

個別のLINE・チャットで報告されても、糸山さん・磯貝が全体を把握できない。
どの案件がどの状態なのか、リアルタイムで見えない。

共通する根本原因

「人が気づく」「人が報告する」に依存している

3 ソリューション — KINTONE × Chatwork 自動アラート

KINTONEのデータを毎朝自動チェック → Chatworkに通知

担当者の「気づき」「報告」に依存せず、システムが自動検知して全員に通知する。
城さんが手動でやっている確認作業を自動化し、漏れをゼロにする。

4 アラート設計 — 2段階アラート

通常アラート — 予防（全員に通知）

問題が起きる前に気づかせる。城さんが日々やっている確認を自動化。

条件アラート内容タイミング
入金予定日の前々日 通常 入金予定リマインド 入金予定日 − 2日
入金予定日の前日 通常 入金予定リマインド（最終確認） 入金予定日 − 1日
日付の逆転・不整合 通常 データ異常アラート 検知時（毎朝チェック）

🚨 緊急アラート — 鈴木さんに通知

問題が既に起きている。即対応が必要。

条件アラート内容判定ロジック
請求書未発行 緊急 請求書未発行アラート 陸送引上日 + 2日経過 かつ 請求書ステータスが「完了」以外
入金遅延 緊急 入金遅延アラート 入金予定日を経過 かつ 入金ステータスが未完了

7 既存の仕組みとの関係
⚠ 重要：既存のExcelチェックシステムは壊しません

現在のExcel入出金予定チェックシステム（磯貝作成・v33）は、KINTONEの日付整合性チェックとして正しく機能しています。
城さん・心ちゃんが資金繰り予測表に使用するフローもそのまま維持します。

役割分担

仕組み役割変更
Excel チェックシステム 日付整合性チェック、資金繰り予測表の元データ 変更なし
Chatwork 自動アラート（新規） 入金遅延の早期検知・通知。「気づかない」「報告漏れ」を防止 追加
資金管理ダッシュボード 入出金の全体可視化・予測・シミュレーション 変更なし

既存の仕組みの上に「通知レイヤー」を追加するだけ

現場のオペレーションは一切変えない

8 導入ステップ
Phase 1 — 準備（1日）
KINTONE APIトークン発行 / Chatwork Webhook URL取得 / 対象アプリ・フィールド確認
Phase 2 — 開発（2-3日）
Python スクリプト開発 / アラート条件実装 / Chatwork通知フォーマット / テスト
Phase 3 — テスト運用（1週間）
磯貝のみに通知 → 内容確認 → 条件調整 → 問題なければ全員に展開
Phase 4 — 本番稼働
GitHub Actions でスケジュール実行開始 / 通知先を全員に拡大
